High-Fidelity MedDRA Bootstrapping and Parameter-Efficient Multi-Task Fine-Tuning of Qwen3-14B for an AI-Led Medical Review Assistant
Clinical Context and Problem Definition
In global pharmacovigilance operations, the processing of Individual Case Safety Reports (ICSRs) forms the basis for safety signal detection, regulatory compliance, and overall public health surveillance.[1, 2, 3] When spontaneous adverse event reports or clinical trial narratives are received, safety departments must process them through a rigorous clinical review workflow.[2, 3] This workflow involves evaluating case seriousness, assigning standardized Medical Dictionary for Regulatory Activities (MedDRA) codes, assessing labeling expectedness against reference safety information, and conducting structured causality assessments to determine the relationship between the suspect drug and the clinical event.[3, 4, 5, 6]
Automating this complex, multi-stage narrative review by training a specialized large language model (LLM), such as Qwen3-14B, requires high-quality, task-specific instruction-tuning datasets.[7, 8, 9] The user's specific challenge involves a custom dataset of 2,000 clinical narratives that contain detailed safety observations but lack standard MedDRA codes.[2, 10] This missing clinical coding represents a major bottleneck.[2] Hand-labeling these narratives is slow, resource-intensive, and prone to inconsistency due to variations in medical training among human coders.[2, 11, 12] On the other hand, relying on zero-shot LLM predictions to generate codes directly from narratives often introduces clinical hallucinations, such as generating non-existent codes, assigning outdated terms, or over-coding minor symptoms.[12, 13, 14]
To solve this problem, a systematic engineering strategy is required to build a programmatic, hybrid, and agentic bootstrapping pipeline.[15, 16] This pipeline uses multi-method candidate retrieval, commercial LLM rank fusion, and closed-loop agentic validation to generate high-confidence MedDRA codes for the 2,000 training instances.[15, 16] By compiling these bootstrapped codes into a multi-task instruction dataset, developers can use Parameter-Efficient Fine-Tuning (PEFT) to train Qwen3-14B into an audit-ready Medical Review Assistant.[8, 17, 18]
Comprehensive Clinical Entity Extraction
Before mapping patient narratives to MedDRA codes, the system must first locate and extract the exact clinical terms within the unstructured text.[15, 19] This initial clinical entity extraction separates the actual adverse event descriptions from surrounding patient history and clinical noise.[11, 15] Selecting an appropriate Named Entity Recognition and Linking (NER+L) tool is critical to ensure high recall and precision during this stage.[19, 20]

Programmatic Bootstrapping Pipeline for MedDRA Label Generation
The core challenge of generating reliable MedDRA codes without manual annotation is addressed using a three-phase programmatic bootstrapping architecture.[15] This design combines rule-based lookup, semantic vector databases, zero-shot rank fusion, and agentic clinical validation.[15, 16]

Phase 4: Closed-Loop Agentic Verification (The MedDCR Framework)
To ensure the bootstrapped codes comply with regulatory standards, the top 10 candidates are processed through a closed-loop multi-agent verification system modeled on the MedDCR architecture.[16] This framework consists of three specialized agents:
The Designer Agent: Formulates a clinical evaluation plan using successful coding strategies retrieved from a historical memory archive.[16] The plan specifies the validation rules and thresholds required for the case.[16]
The Coder Agent: Compiles the Designer's plan into an executable programmatic script.[16] The Coder programmatically queries the active MedDRA files to verify that the selected terms are currently active, discarding any non-current or retired codes.[2, 16, 26] It also includes a self-fixing loop: if the generated code throws an exception during execution, the Coder parses the traceback error and automatically regenerates a corrected version of the script.[16]
The Reflector Agent: Acts as a clinical safety auditor, evaluating the Coder's proposed terms against the official, ICH-endorsed MedDRA Term Selection: Points to Consider (MTS:PTC) guidelines.[10, 11, 16]
The Reflector Agent applies several clinical rules to ensure data quality:
Diagnosis vs. Symptoms Rule: If the narrative describes a definitive clinical diagnosis accompanied by its characteristic signs and symptoms (e.g., "acute myocardial infarction" presenting with "chest pain, diaphoresis, and dyspnea"), the Reflector selects only the single term for the diagnosis ("Myocardial infarction") and discards the individual symptom terms to prevent over-coding.[2, 11, 27]
Multi-Axial Priority Mapping: Because a single Preferred Term (PT) can map to multiple System Organ Classes (SOCs), the Reflector enforces primary SOC priority rules to ensure standard reporting structure.[12, 26] For example, if a term links to more than one exception, it resolves the primary SOC using the mandated hierarchical priority order: 1st: Congenital, familial and genetic disorders; 2nd: Neoplasms benign, malignant and unspecified; 3rd: Infections and infestations.[12, 26]
Exclusion of Non-Events: The Reflector identifies and filters out candidate terms associated with family history, negative findings, or ruled-out/hypothetical events.[16, 22]
If a case fails these validation rules, the Reflector returns clinical feedback to the Designer and Coder, triggering a re-evaluation.[16] Once the agents reach consensus, the selected 8-digit numeric MedDRA code is committed as the gold-standard label for that training instance.[2, 16, 26]
Custom Dataset Synthesis and Multi-Task Instruction Engineering
Once the programmatic bootstrapping pipeline has generated reliable MedDRA codes for the 2,000 narratives, the data is structured into a multi-task instruction-tuning dataset.[8, 9] The dataset uses a standard format with three fields: instruction (the system prompt), input (the unstructured clinical safety narrative), and output (a structured JSON payload).[8]
To train the Qwen3-14B model to act as a comprehensive Medical Review Assistant, the structured JSON schema integrates four clinical review tasks into a single output [3, 8]:
Seriousness Assessment: Evaluating whether the event meets regulatory seriousness criteria under the ICH E2D guidelines (e.g., resulting in death, life-threatening, requiring hospitalization, causing disability, or representing a medically significant event).[3]
MedDRA Coding: Extracting and mapping the clinical events to their corresponding Lowest Level Term (LLT), Preferred Term (PT), primary System Organ Class (SOC), and 8-digit numeric codes.[2, 26, 28]
Labelling Status Evaluation: Cross-referencing the drug's package insert (retrieved via the DailyMed API using the drug's National Drug Code (NDC) or SETID) to determine if the event is "Expected" or "Unexpected".[5, 29, 30]
Causality Assessment Support: Calculating causality ratings using both the objective Naranjo Adverse Drug Reaction Probability Scale and the WHO-Uppsala Monitoring Centre (WHO-UMC) system.[4, 31, 32]

Parameter-Efficient Fine-Tuning on Qwen3-14B
Model Architecture and Hardware Considerations
Qwen3-14B is a dense, causal decoder-only transformer with 14.8 billion parameters.[7] The model's architecture consists of 40 layers, utilizing Grouped Query Attention (GQA) with 40 query heads and 8 key/value heads to improve inference throughput.[7] It incorporates standard features such as SwiGLU activations, RMSNorm for pre-normalization, and Rotary Positional Embeddings (RoPE).[7]
A model of this scale presents significant memory challenges during training.[34] While standard inference can run on consumer hardware, full-parameter fine-tuning requires tracking model weights, gradients, and optimizer states (such as the two momentum terms per parameter in AdamW), multiplying the memory requirement by 12Г— to 16Г—.[34] This raises the minimum VRAM requirement for full fine-tuning to over 160 GB.[34] To train Qwen3-14B on accessible hardware, developers must choose between standard Low-Rank Adaptation (LoRA) and 4-bit Quantized LoRA (QLoRA).[34, 35, 36]

Under QLoRA, the base model is loaded in 4-bit precision (NF4), while the trainable LoRA adapters run in 16-bit precision.[35] Paged optimizers are enabled to prevent out-of-memory errors by offloading gradient memory spikes to the CPU.[34, 35] While QLoRA reduces the memory footprint, it introduces several limitations: it does not support weight merging (requiring adapters to be loaded separately at inference), cannot train embedding layers (preventing the model from learning new custom tokens), and must use FP16 precision.[35, 36]
For organizations with access to professional hardware (such as an A100 80GB), standard LoRA in FP16/BF16 is the preferred option because it supports weight merging, enables training of embedding layers (wte and lm_head) to learn ChatML special tokens, and provides faster training speeds.[36]

Learnings, Operational Pitfalls, and Future Directions
Developing this programmatic bootstrapping and fine-tuning pipeline yields several technical and operational lessons:
Core Learnings and System Dynamics
Task Interdependence: Medical review tasks are highly interconnected.[33] For example, assigning an incorrect MedDRA code directly compromises the downstream labeling evaluation, which can lead to safety signals being missed.[1, 33]
The Value of Multi-Agent Verification: Using a multi-agent validation loop (such as the MedDCR Designer-Coder-Reflector framework) reduces code hallucinations by over 10% compared to standard zero-shot LLM predictions.[13, 16] The system's ability to self-correct by parsing traceback errors is critical for data quality.[16]
Adapter Efficiency: Fine-tuning Qwen3-14B using LoRA with target modules applied to all linear layers provides a balanced combination of rapid learning convergence, low memory usage, and preserved pre-trained medical knowledge.[18, 36, 39]
Critical Failure Modes and Mitigations
Dictionary Version Drifts: MedDRA undergoes bi-annual updates (released every March and September).[2] If the training dataset is mapped to an older dictionary version, the fine-tuned model will generate outdated terms, which can lead to regulatory submission failures.[2, 12] Mitigation: Implement structured version-control and programmatically update the memory archives and vector database embeddings with every new MedDRA release, followed by targeted adapter updates.[2]
Multi-Axial Code Ambiguity: A Preferred Term (PT) can map to multiple System Organ Classes (SOCs), which can lead to classification errors when generating structured JSON payloads.[12, 26] Mitigation: The Reflector agent must strictly enforce primary SOC priority rules (e.g., Congenital, Neoplasms, and Infections exceptions) to ensure consistent, compliant mappings.[12, 26]
Low-Frequency Code Hallucinations: LLMs frequently struggle with rare adverse events that appear infrequently in the training data, leading to a drop in precision.[13, 14] Mitigation: Utilize data augmentation techniques and incorporate few-shot in-context learning examples of rare events within the Designer AgentвҖҷs memory archive.[13, 16]
Future Strategic Directions
Continuous Multi-Agent Workflows: Future architectures should allow the multi-agent system to update its own workflows dynamically based on changing regulatory guidelines, rather than relying on a static instruction set.[16]
Global Multi-Lingual Expansion: MedDRA is inherently multi-lingual, and global pharmacovigilance requires processing safety reports in multiple languages.[7, 12, 28] Future datasets should incorporate multi-lingual clinical narratives, taking advantage of Qwen3вҖҷs extensive pre-training across 119 languages.[7, 28]
Integration of WHODrug Coding: To build a more comprehensive pharmacovigilance assistant, future datasets should include drug name mapping to WHODrug codes alongside MedDRA adverse event coding.[28] This would allow the system to analyze concomitant medications and support more robust causality assessments.[28]
